# Contributing

We accept PRs, not pitch decks.

## The Deal

If you contribute a high-quality engineering template, guide, or fix to this repo, you get a **fast-track interview at MiraclePlus**. Not a guaranteed investment — a guaranteed conversation with someone technical who can evaluate your work.

Why? Because the kind of person who writes production-grade open source is exactly who we want to back.

## What We're Looking For

### Templates (highest priority)

Production-ready engineering templates that solve real problems researcher-founders face:

- Model optimization pipelines (quantization, distillation, pruning)
- Data pipelines (labeling, cleaning, augmentation at scale)
- Monitoring and observability for ML services
- CI/CD for ML models
- Multi-modal serving (text + image + audio)
- Edge deployment (mobile, embedded)
- RAG pipelines
- Fine-tuning infrastructure

### Guides

Practical guides from founders who've been through it:

- How to negotiate with cloud providers
- Hiring your first engineer (when you're a researcher)
- Pricing your AI product
- Building in China, selling globally (and vice versa)
- Navigating AI regulations

### Improvements

Bug fixes, performance improvements, documentation fixes to existing templates.

## Quality Bar

This isn't a blog. We have a high bar:

1. **Code must work.** Not "should work" — actually tested, actually runs.
2. **No vendor lock-in.** Templates should work with standard tools.
3. **Comments explain WHY, not WHAT.** We can read code. Tell us why you made the choices you made.
4. **Opinionated is good.** Don't give 5 options. Pick the best one and explain why.

## How to Contribute

1. **Check existing issues** for template requests or discussions.
2. **Open an issue first** if you're proposing something new. Get buy-in before writing code.
3. **Fork and PR.** Standard GitHub workflow.
4. **Include a README** in your template explaining what it does and how to use it.
5. **Include tests.** If it's code, it needs tests.

## PR Template

```
## What

One sentence on what this adds.

## Why

Why does this matter for researcher-founders?

## How to test

Steps to verify this works.
```

## Code Style

- Python: Follow PEP 8. Use type hints. We're not strict about formatters.
- YAML: Use comments liberally. These configs are documentation.
- Markdown: No fluff. Get to the point.

## Response Time

We review PRs within a week. If it's been longer, ping us.
